# APEX XI ‚Äî StatsBomb Historic Batch Pipeline (Design Spec)

**Date:** 2026-07-05
**Status:** Approved (pending spec review)
**Scope:** Sub-project 1 of the APEX XI platform ‚Äî the **batch layer** of the Lambda architecture. Later sub-projects (own specs): speed layer (ScraperFC live WC 2026), serving API, dashboard frontend.

## Problem

The APEX XI dashboard (wireframe: `docs/APEX XI - Football Analytics v2.html`) has a **Historic mode** with a **club-selector dropdown** and rich analytics ‚Äî player/team radars (xG/xA, percentiles), team stats, and shot maps. This spec builds the batch pipeline that turns StatsBomb open event data into dashboard-ready serving tables for that mode.

## Key Decisions (locked)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Data source | **StatsBomb open data** via `statsbombpy` | User-specified; free; event-level with built-in xG |
| Dataset | **La Liga 2015/16** (`competition_id=11`, `season_id=27`) | **Verified**: the ONE full La Liga season in open data ‚Äî 380 matches, **all 20 clubs**, each plays 38. Every other La Liga season is Bar√ßa-only. Satisfies "all clubs" + "La Liga" + "StatsBomb" simultaneously. Wireframe's dropdown clubs match this set exactly. |
| Architecture | Layered batch pipeline (Approach A): ingest ‚Üí clean ‚Üí model ‚Üí serve | Independently testable stages; immutable cleaned master; matches the Lambda batch‚Üíserving pattern; no heavy framework |
| Compute | Python + pandas + statsbombpy | ~1.3M events (laptop-scale); Spark/Hadoop would be overkill |
| Serving store | **Parquet + DuckDB** | Embedded, no server, portable; ideal for analytical queries by the future API |
| Serving outputs | `player_season` (+percentiles), `team_season`, `shots` | Powers Players, Teams, Compare, Standings, shot maps ‚Äî most of Historic mode |
| Orchestration | Small `argparse` CLI (`apex ingest|clean|model|serve|build`) | YAGNI on Airflow/Prefect at this scale |

## Architecture & Data Flow

```
statsbombpy  (La Liga 2015/16: competition_id=11, season_id=27 ‚Äî 380 matches, 20 clubs)
        ‚îÇ  apex ingest   (ONLY stage that touches the network; idempotent cache)
        ‚ñº
data/raw/ matches.parquet + events/<match_id>.parquet
        ‚îÇ  apex clean
        ‚ñº
data/processed/events.parquet     ‚Üê immutable cleaned MASTER (one tidy row per event)
        ‚îÇ  apex model   (3 independent transforms read the master)
        ‚îú‚îÄ‚îÄ data/serving/player_season.parquet   (+ league-wide position-group percentiles)
        ‚îú‚îÄ‚îÄ data/serving/team_season.parquet
        ‚îî‚îÄ‚îÄ data/serving/shots.parquet
        ‚îÇ  apex serve
        ‚ñº
data/serving/apex.duckdb   ‚Üê 3 tables materialized from serving Parquet (future API queries this)
```

**Invariants:**
- Only `ingest` uses the network. `clean`/`model`/`serve` run offline from cached files.
- `events.parquet` is the immutable master; `model` recomputes serving views from it without re-fetching or re-cleaning (the Lambda batch layer property).
- Every serving table carries a **club/team dimension** so the dashboard's club dropdown filters it.

## Stage 1 ‚Äî Ingest (`apex ingest`)

- `sb.competitions()` sanity-check that `competition_id=11, season_id=27` exists.
- `sb.matches(competition_id=11, season_id=27)` ‚Üí 380 match rows ‚Üí `data/raw/matches.parquet`.
- For each `match_id`: if `data/raw/events/<match_id>.parquet` absent (or `--force`), `sb.events(match_id=‚Ä¶)` ‚Üí write it. Skip existing ‚Üí idempotent, re-fetch-safe.
- Per-match HTTP failure: retry a few times, then skip and record; report `N succeeded / M failed` at the end. Never write a partial match file (write to temp, then rename).
- `statsbombpy`'s open-data attribution message is expected, not an error.

## Stage 2 ‚Äî Clean (`apex clean`)

- Read all `data/raw/events/*.parquet` + `matches.parquet`.
- Concatenate; stamp `match_id, competition_id, season_id`; join match context (`home/away`, `opponent`, `match_date`, `team_id`).
- **Flatten & type:** `location` `[x,y]` ‚Üí `location_x/location_y`; pass/carry/shot end locations likewise; cast numeric/bool/category; parse `minute/second`.
- **Column set** (lean but complete for the 3 models):
  - Identity/context: `id, type, team, team_id, player, player_id, position, possession, possession_team, play_pattern, under_pressure, duration, minute, second, period`.
  - Pass: `pass_length, pass_end_x, pass_end_y, pass_outcome, pass_recipient, pass_cross, pass_shot_assist, pass_goal_assist, key_pass_id`.
  - Shot: `shot_statsbomb_xg, shot_end_x, shot_end_y, shot_outcome, shot_type, shot_body_part, shot_key_pass_id`.
  - Defensive-action markers for PPDA: pressures, tackles (duels), interceptions, ball recoveries (derived from `type`).
- **Validation (fail loudly):** unique event `id`; non-null `type, team_id, match_id`; every `type == "Shot"` row has non-null `shot_statsbomb_xg`.
- Write `data/processed/events.parquet` (immutable master).

## Stage 3 ‚Äî Model (`apex model`)

### `player_season.parquet` ‚Äî one row per (player_id, team_id, season)
- **Minutes** derived from Starting-XI + Substitution events per match; `appearances` from matches featured. Eligibility threshold `MIN_MINUTES = 450` gates percentile inclusion.
- **Raw totals + per-90** (per-90 = total / minutes √ó 90):
  - Attacking: goals, assists, `xg` (Œ£ `shot_statsbomb_xg`), `xa` (Œ£ xG of shots assisted via key-pass‚Üíshot link), shots, shots_on_target.
  - Possession: passes, pass_completion_pct, progressive_passes, passes_into_final_third, key_passes, carries, progressive_carries, dribbles_completed.
  - Defending: pressures, tackles, interceptions, ball_recoveries.
- `primary_position` (most-played) ‚Üí `position_group ‚àà {GK, DEF, MID, FWD}` via `config.POSITION_MAP`.
- **Percentiles:** for each per-90 metric, `percentile_<metric>` = rank (0‚Äì100) within the player's `position_group` across all 20 clubs' eligible players. GK excluded from outfield radars.
- Columns: identity (`player_id, player, team_id, team, season, primary_position, position_group, minutes, appearances`) + raw + per-90 + `percentile_*`.

### `team_season.parquet` ‚Äî one row per (team_id, season), all 20 clubs
- Standings: `matches, wins, draws, losses, points, gf, ga, gd` (real full-league table from match results).
- Style/quality: `xg_for, xg_against, xg_diff, possession_pct, ppda` (opponent passes √∑ own defensive actions in the pressing zone, opp. half), `pass_completion_pct, progressive_passes, shots_for, shots_against`.
- Team-level `percentile_*` across the 20 clubs for the Teams radar.

### `shots.parquet` ‚Äî one row per shot (all clubs)
- `shot_id, match_id, team_id, team, player_id, player, minute, location_x, location_y, shot_statsbomb_xg, outcome, body_part, shot_type, play_pattern, under_pressure`.
- Powers shot maps + xG timelines; filterable by club / player / match.

## Stage 4 ‚Äî Serve (`apex serve`)

- Create/replace `data/serving/apex.duckdb` with three tables materialized from the serving Parquet:
  `CREATE OR REPLACE TABLE player_season AS SELECT * FROM read_parquet('data/serving/player_season.parquet');` (and team_season, shots).
- Self-contained DB file the future serving API queries. (Views over Parquet were considered; materialized tables chosen for a stable, portable serving artifact.)

## CLI (`apex/cli.py`)

`argparse` subcommands calling the stage functions: `ingest [--season N] [--force]`, `clean`, `model`, `serve`, `build` (= ingest‚Üíclean‚Üímodel‚Üíserve). Each stage function is importable and independently runnable.

## Error Handling

| Situation | Behavior |
|-----------|----------|
| `ingest` per-match HTTP failure | retry (few attempts) ‚Üí skip ‚Üí count; temp-write+rename (no partial files) |
| `clean` schema violation (dup id, null key, shot w/o xG) | raise immediately with the offending detail |
| `model`/`serve` | assume validated master; log row counts per output |
| Missing prior-stage output | clear error telling the user which stage to run first |

## Testing (TDD)

- `tests/fixtures/` ‚Äî 2‚Äì3 real recorded StatsBomb matches (raw events) for deterministic tests.
- `test_clean` ‚Äî location flattening, dtypes, unique-id, validation raises on injected bad rows.
- `test_player_season` ‚Äî goals/xG/per-90 aggregates, xA via key-pass link, position grouping, percentile rank within group, minutes threshold.
- `test_team_season` ‚Äî points/W-D-L from results, xG for/against, PPDA, possession%.
- `test_shots` ‚Äî correct shot rows/fields.
- `test_build` ‚Äî integration: run pipeline on a 1‚Äì2 match subset (from fixtures, no network) ‚Üí 3 serving tables exist and `apex.duckdb` is queryable.

## Out of Scope (later sub-projects)

Speed layer (ScraperFC live WC 2026), serving REST API, dashboard frontend, per-match/pass-network/set-piece tables, non-La-Liga competitions.

## Risks / Open Items

- **Minutes-played** isn't a native StatsBomb field; derived from Starting-XI/Substitution events. Documented, unit-tested; small edge cases (red cards, extra time) tolerated.
- **xA attribution** relies on the shot‚Üíkey-pass link present in StatsBomb events; passes without a resulting shot contribute 0.
- **statsbombpy availability**: open data is served from GitHub; `ingest` must tolerate transient failures (handled above). First plan step verifies `statsbombpy` installs and `season_id=27` is reachable.
- **PPDA zone definition** uses opponent passes in their own 60% of the pitch √∑ our defensive actions there ‚Äî a standard but choice-laden definition; documented in `config.py`.
